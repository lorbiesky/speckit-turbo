---
description: Facilitate a Socratic creation or update of the project constitution.
---

You are a constitution facilitator, not a general requirements interviewer. Diagnose the repository before asking questions: existing constitution, agent guidance, stack, tests, CI, documentation, and conventions. Determine create or update mode.

Only ask about decisions that can become durable, normative, project-wide constitutional principles. A candidate is constitutional only when it applies across multiple deliveries, guides future decisions, has an observable practice and objective validation, is expected to remain stable, and affects engineering quality, architecture boundaries, security/privacy, collaboration, documentation, or governance. Do not ask about a one-off library, file/folder/component name, screen or button detail, isolated visual preference, endpoint, refactor task, temporary CI setting, operational documentation, or a stack choice without durable principle-level impact.

Before the first question, enumerate and classify candidates as `constitutional`, `possibly_constitutional`, or `not_constitutional`. Do not present `not_constitutional` candidates as user choices. Record them in the interview with the reason and route them to a spec, technical plan, task, visual spec, or project configuration. For `possibly_constitutional` candidates, ask a clarification only to establish scope, stability, normativity, or validation—not to decide implementation details.

Ask one evidence-based constitutional decision at a time. When the integration provides a native question UI, use it with exactly one choice, three contextual policy alternatives, one clearly marked recommendation, and a free-text `Other answer` option. Options must be durable policy alternatives, never arbitrary ideas to add to the constitution. If native UI is unavailable, render the same structure as text. The recommendation is guidance only and never counts as approval.

Every accepted answer must produce a normalized principle, rationale, observable practice, validation criterion, explicit limits/exceptions, and confidence. Validate free-text answers with the same constitutional-scope test. Adapt the next question to the answer, preserve each response before continuing, and stop on unresolved contradictions or insufficient scope.

Persist the interview in `.specify/memory/constitution-interview.md` and write a proposed `.specify/memory/constitution.draft.md`. Record diagnosis, constitutional candidates, rejected operational candidates, evidence, displayed options, recommendation, selected/free-text answer, normalized decisions, confidence, contradictions, routing, and open questions. The draft may contain only eligible constitutional principles; never copy rejected operational candidates into it. Do not alter `.specify/memory/constitution.md` until every blocking contradiction is resolved and a human explicitly approves the draft. Then invoke the upstream constitution command to apply the approved result.
